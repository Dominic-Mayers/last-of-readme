The split you identify is necessary, and the reason is precise.

The installer builds the dependency graph from constraint declarations. For it to do this without knowing anything about Last of Readme, the dependency-determining part of each constraint must be expressible in a language the installer understands. If the installer had to understand LOR semantics to know what depends on what, it would not be a universal tool — it would be an LOR-specific tool that happens to be called an installer.

In constraint satisfaction terms, the split corresponds to:

Scope — which configuration variables must be established before this constraint can be checked. The installer reads this to build the dependency graph. The structure is installer-general; the variable names may be application-specific, but the installer does not need to understand their meaning, only their relationships.
Predicate — the actual check on those variable values once they are established. This is opaque to the installer; it just executes it when the prerequisites are satisfied.
The scope is what makes the installer universal. The installer reasons about scopes, not predicates. The application declares both, but the installer uses only the scopes for orchestration.

The strategy-relative nature of dependencies also fits here cleanly. The scope of a constraint depends on the solving strategy chosen. "Determine the repository API URL by asking the user directly" has an empty scope. "Determine it by deriving from the selected remote" has scope {selected_remote}. Same constraint, different strategy, different position in the dependency graph. The scope is not intrinsic to the constraint — it is relative to how the installer chooses to satisfy it. The predicate, by contrast, does not change with strategy.

So the full picture of a constraint on a component configuration is:


(scope, predicate, strategy)

scope:     variables that must be established first — installer-general structure
predicate: the check on those values — LOR-specific, opaque to the installer  
strategy:  how candidate values are obtained — may be installer-general or LOR-specific
The dependency-determining part is the scope. It must be general enough for the installer to process uniformly across all applications. The predicate and the application-specific parts of the strategy are what the LOR layer contributes.
